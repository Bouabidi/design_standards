Forms and CRUD

Consistent actions and feedback on forms and form actions are essential to maintain the application's usability. When users perform CRUD actions, they should understand what is happening and anticipate change resulting from thier action.

Action States

  • Create case: it is a button (create, add, etc.) that either flips a div, navigates to a form page or spawns a modal.
  • Edit case: it is a button (edit, change, icon, etc.) that either flips a div, navigates to the form page or spawns a modal.
    • If required fields exist then the “save” button should be disabled until they have been populated. Once the user has populated required input, the button is active.
    • If no required fields exist, “save” button can be active on initial load.
  • Delete case: it is a button (delete, remove, icon, etc.) that deletes an item from the system (a least from the user’s perspective). There are two delete interactions possible:
    • Non-prompted, unconfirmed delete – where clicking button instantaneously removes the content. This type of delete interaction is best for “low cost” items – content that is easily replicated by the user.
      • Examples: demographics, comments, tags, announcements, etc.
    • Prompted delete confirmation – where an alert is thrown to confirm this action is what was intended. A delete confirmation should be used for “high cost” item – content that is more difficult to replicate.
      • Examples: admin configuration pages, events, volunteer opportunities, communities, libraries, blogs, etc.
