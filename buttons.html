Buttons

Buttons indicate actions a user may take. Their consistent display, position and labeling is essential to a positive user experience.

Guidelines

  • Use appropriate color for a given action, shown below
  • Button Text should be concise and state explicitly the action the button represents:
    • "Save" - universal system submit action text. Always use "Save" for action modals unless task documentation/requires say otherwise
    • "Cancel" - universal system abort text. Always use "Cancel" for action modals unless task documentation/requires say otherwise.
    • "Close" - universal system close text for non-action modals. Always use "Close" for non-action modals (informational modals) unless task documentation/requires say otherwise.
  • Button placement depends on the type of action container is at play:
    • For modals and form pages, action buttons are always aligned left, with the positive action in the left most position.
    • For list pages, list-level action buttons (Add, Export, etc) always appear above the sort & filter option, right justified.
  • Multi-action buttons (split buttons) are where multiple actions exists for a given item. In this case, the dominant action is the the button and the secondary, tertiary actions, etc. are stacked up in the dropdown.
  • Display of profile photo honors user configurations.

Examples

Button Colors

Button colors provide visual cues for users. In general, all buttons should adhere to the rules below.

Save - blue for core system funcitonality: save, edit, submit, reply, etc.

Add - green for adding content, accepting an action, etc.

Delete - red for negative or delete actions.

Cancel - white or light grey for cancel and in cases where actions need to be visually minimized.

Save - disabled button.

Disabled State

For cases where the action is not available or ready, use the disabled state to indicate the presence of the action. This is common for forms and actions that require user input. An example is a login form that requires both a UID and Password. Until both fields are "dirty", the button is disabled. Once the user has entered something in both fields, the button is enabled.

More on CRUD Buttons (create, read, updated and delete).

Consistent actions and feedback on forms and form actions are essential to maintain the application's usability. When users perform CRUD actions, they should understand what is happening and anticipate change resulting from thier action. For all each of the states below, please follow instructions and design information in the task definition for more granular instructions.

  • Create : button (create, add, etc.) that either flips a div, navigates to a form page or spawns a modal - depends on the use case.
    • Use bootstrap class="btn btn-success" for create actions.
    • Example: Add
  • Save : button should be present a disabled state (add "disabled" class to anchor) until required form fields have been populated with data. This keeps the user from submitting an incomplete form.
    • Use bootstrap class="btn btn-primary" for Save actions.
    • Example, live button: Save
    • Example, disabled button: Save
  • Edit : button (edit, change, icon, etc.) that either flips a div, navigates to the form page or spawns a modal.
    • Example: Edit
  • Delete: button (delete, remove, icon, etc.) that deletes an item from the system (a least from the user’s perspective). There are two delete interactions possible:
    • Example: Delete
    • Non-prompted, unconfirmed delete – where clicking button instantaneously removes the content. This type of delete interaction is best for “low cost” items – content that is easily replicated by the user.
      • Examples: demographics, comments, tags, announcements, etc.
    • Prompted delete confirmation – where an alert is thrown to confirm this action is what was intended. A delete confirmation should be used for “high cost” item – content that is more difficult to replicate.
      • Examples: admin configuration pages, events, volunteer opportunities, communities, libraries, blogs, etc.
  • Cancel: button is used for all process forms and modals. There are two delete interactions possible:
    • Example: Cancel
